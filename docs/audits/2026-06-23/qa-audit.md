# QA Audit — nomad.md (Step 6 prep)

> Auditor: QA committee member
> Date: 2026-06-23
> Scope: Steps 1-5 + Step 6 readiness

## Verdict

**No.** The test suite is well-organised and contains genuine, contract-level assertions — not just "no throw" smoke checks — but the coverage is **window-dressed at three layers**:

1. `src/lib/services/integrity.ts` has **no dedicated test file**. `verifyIntegrity` (lines 51-55, the function that detects tampering) is at 16.66% branch coverage and is **only indirectly exercised** through the parser. The state-of-the-art doc claims the integrity layer is tested; it is not.
2. `src/lib/adapters/remote-git.ts` sits at 30.20% statements / 28.16% branches / 22.22% functions. The `fetchSubtree` flow (lines 57-275, 291-536) — the **PAT-consuming path** the entire security posture rests on — has zero tests of its own. The mode test that claims to cover PAT hygiene (`tests/state/mode.test.ts:244`) only inspects `Object.keys(store)`; it never calls `openRemote`, never invokes `fetchSubtree`, never asserts that a PAT flows into `onAuth` and is dropped on return.
3. There are **zero Svelte component tests and zero Playwright E2E specs**, despite the project carrying `@vitest/browser-playwright`, `playwright`, and `vitest-browser-svelte` in devDependencies and the `client` Vitest project wired to a Chromium browser. The `client` project only runs `tests/adapters/handle-store.test.ts` — IndexedDB against the real handleStore. `vitest-browser-svelte` is **not imported anywhere** in the codebase.

The biggest single gap is the **`openRemote` + `fetchSubtree` + PAT-hygiene contract**, which is the cornerstone of NFR-2, and the audit committee is being told via two prior audits and the state-of-the-art that "the PAT hygiene is covered." It is not. See `tests/state/mode.test.ts:235-256` — the "PAT hygiene" describe block contains exactly **two tests**, neither of which ever calls `openRemote`. This is the single most consequential window-dressing in the entire test suite.

## Edge cases missing (top 10)

For each:

- **Description** — what scenario
- **Why it matters** — what bug it would catch
- **Where to add** — file path / test file
- **Effort** — S/M/L

1. **Empty `.nomad.md/` directory on first run** (FR-11 wizard path)
   - **Why**: A user opens a fresh folder; the wizard needs to detect "no config, no templates, no issues" and offer to bootstrap. `config.ts:load()` handles missing config; `templates.ts:load()` handles missing dir; `issues.ts:load()` has no missing-dir handler at all (it would throw on `listDirectory`). The chain is not tested.
   - **Where**: `tests/state/integration.test.ts` — add `describe('first-run bootstrap')` block.
   - **Effort**: M

2. **PAT with leading/trailing whitespace** (`"  ghp_…  "`)
   - **Why**: `mode.ts:openRemote` passes `pat` straight to `fetchSubtree` which passes to `brandPat`. A whitespace-padded PAT would be branded but fail server-side auth, confusing the user. There is no `.trim()` and no test.
   - **Where**: `tests/state/mode.test.ts` — add `it('strips PAT whitespace before brand')`.
   - **Effort**: S

3. **Two issues with the same title** — `nextIssueId` collision logic
   - **Why**: `src/lib/services/slugs.ts` (untested lines 23, 46) likely contains collision-counter logic. Two issues titled "Foo" would collide in `buildIssueFilename`. The current tests only create one issue per title.
   - **Where**: `tests/services/slugs.test.ts` (does not exist; create it).
   - **Effort**: S

4. **Unicode / emoji in filenames** — `"重要-🚀.md"`
   - **Why**: The `nextIssueId` + `buildIssueFilename` slug logic is not tested against non-ASCII. If the slugifier drops Unicode silently, two distinct issues could collide. No test exists.
   - **Where**: `tests/services/slugs.test.ts`.
   - **Effort**: S

5. **Super-long issue body** (>1 MB Markdown)
   - **Why**: `local-fs.ts` enforces a `MAX_FILE_SIZE = 10 MiB`. `serializeIssue` produces the body; if a section has 9.9 MiB of text, the resulting file is at the limit. No test exercises the boundary or rejection.
   - **Where**: `tests/adapters/local-fs.test.ts` and `tests/services/serializer.test.ts`.
   - **Effort**: M

6. **Integrity hash mismatch recovery path** — manually-corrupted file loaded via real `loadIssues`
   - **Why**: The current test (`tests/state/issues.test.ts:383-401`) manually constructs the corrupted file and only asserts `integrityWarnings.length === 1`. It does not assert the recovery flow (save re-computes; verifyIntegrity then returns true). A regression in `verifyIntegrity`'s prefix check would not be caught.
   - **Where**: New file `tests/services/integrity.test.ts`.
   - **Effort**: M

7. **Cycle in relations** — already partially tested, but no test exercises `depends_on` cycle, only `blocks` cycle (`tests/services/validator.test.ts:71-110`).
   - **Why**: `validator.ts:detectCycles` uses a strict set `STRICT_RELATION_TYPES = ['parent', 'child', 'blocks', 'depends_on']`. The tests only use `blocks`. A regression that broke `depends_on` would not be caught.
   - **Where**: `tests/services/validator.test.ts` — add a `parent → child → parent` cycle test.
   - **Effort**: S

8. **Two `save(id)` calls where the second arrives after the first completes**
   - **Why**: The current test (`tests/state/issues.test.ts:257-275`) fires both `save()` calls **before either resolves**. A regression where the second call's promise is not equal to the first's when the second arrives **after** settle would not be caught. The lock should still kick in (i.e. fresh `doSave`, not join the resolved one).
   - **Where**: `tests/state/issues.test.ts:257` — add `await p1; const p2 = save(); ...`.
   - **Effort**: S

9. **`bootstrap()` vs `signOut()` race** — call `bootstrap` and `signOut` "concurrently" with the same active handle
   - **Why**: Could leak the handle to recent list, or fail to clear mode. Plan §B.10.1 covers it as a Step 6 acceptance test.
   - **Where**: `tests/state/mode.test.ts` — new `describe('bootstrap × signOut race')`.

## Test-quality red flags

1. **`tests/state/mode.test.ts:235-256` — "PAT hygiene (NFR-2)"** — The describe block name claims NFR-2 coverage. The two tests inside (`expect(keys.some((k) => k.toLowerCase().includes('pat'))).toBe(false)`) **only enumerate `Object.keys(store)`**. The store's public methods are `bootstrap`, `openLocalFolder`, `switchFolder`, `openRemote`, `signOut` — none contain "pat". The test would pass if `openRemote` stored the PAT in a module-level variable or even attached it to a new property called `_patScope` (which it does — and **that** property IS the storage, so the test is internally inconsistent). **This is misleading coverage: a future refactor that accidentally retained the PAT in `_patScope` after `signOut` would pass this test unchanged.**

2. **`tests/state/issues.test.ts:257-275` — concurrent save** — The test fires two `save(1)` calls back-to-back and asserts `p1 === p2`. It does NOT exercise the case where `p1` has already resolved before `p2` is called. A regression where the second call becomes a new `doSave` (instead of joining) would be **silently correct** in this test, but wrong in production.

3. **`tests/state/issues.test.ts:407-437` — load supersede** — Uses `current = fs2` to swap the adapter mid-flight. This relies on the **adapter-provider closure being called fresh each time**, which the test doesn't assert. A future refactor that cached the adapter at factory time would still pass this test only because `load()` is called twice (the first being aborted). The test conflates "abort works" with "second load wins"; a clean test would assert `loadController.abort` was called.

4. **`tests/state/integration.test.ts:142-236` — full CRUD journey** — The test creates `mode = createModeStore(...)` and immediately calls `bootstrap()`. It then **never uses `mode` again** — `mode.localAdapter` is `null`, so the `config`/`templates`/`issues` stores use a separate `() => fs` provider. The test claims to exercise "the wiring type-checks and the store constructs" (line 145). It is **not** an integration test of mode+data; it is a unit test that the integration compiles. A future contributor reading the test name would believe the mode store participates in the wiring; it doesn't.

5. **`tests/services/validator.test.ts:42-62` — `makeIssue` fixture** — Always populates `creationDate` and `updatedDate` to `'2026-01-01'`. The validator has explicit branches for empty `creationDate` (line 127), empty `updatedDate` (line 130), empty `author` (line 133) — none of which are reached by any test. Three branches of `validateIssue` are **dead by fixture design**.

6. **`tests/adapters/local-fs.test.ts`** (lines 1-735) — installs a mock `window` and exercises `LocalFsAdapter` end-to-end. This is good. BUT the tests do NOT cover:
   - `writeTextFile` when the target path is **an existing directory** (lines 183-194 of source) — the explicit `AdapterValidationError` arm.
   - `moveFile` **across different parents** (lines 308-313 of source) — only same-parent moves tested.
   - `writeTextFile` exceeding `MAX_FILE_SIZE` (lines 197-202 of source) — no test at 10 MiB + 1.
   - `pick()` rejection on cancel — `cause.name === 'AbortError'` → `FsaPermissionError`. No test.

7. **`tests/state/editor.test.ts:199-208` — `patchField` no-op when no draft** — Asserts `expect(stores.editor.draft).toBeNull()` and `expect(stores.editor.isDirty).toBe(false)`. Good. But the test does NOT verify that calling `patchField('title', 'B')` with `undefined` is a no-op when there's no draft (the editor's branch at line 151 of source: `if (value === undefined) return;`). A regression that flipped the guard would not be caught.

8. **`tests/services/integration.test.ts`** — uses `MemoryFsAdapter` only. It never exercises the `LocalFsAdapter` happy path or any cross-adapter parity test. A regression that breaks the FSA path but not the memory path would not be caught here. (The `local-fs.test.ts` covers the adapter in isolation, but not the **end-to-end** wiring through the services.)

9. **`tests/state/config.test.ts:78-99` — supersede** — Same pattern as issues: relies on closure-over-current-adapter. Same fragility (see #3).

10. **`tests/state/issues.test.ts:553-569` — applyPatch reference identity** — Asserts `expect(heldRef?.['severity']).toBe('low')` after the patch adds `{ priority: 'p1' }`. The contract here is "deep-merge of `customFields` preserves unmentioned keys", but the test does NOT assert what happens when the patch is `{ severity: undefined }` (i.e. deletes a key). A regression that overwrites instead of merging would pass this test.

11. **Mutation spot-check results** — applying obvious mutations to the source:
    - **`issues.ts:282-299` `update()`** — change `if (!snapshots.has(id))` to `if (snapshots.has(id))` (negate). Test `tests/state/issues.test.ts:521-547` would **still pass** because the snapshot would still be stored; only the "fresh vs reused" semantic would break. **Mutation undetected.**

But: the test does NOT exercise a flow where `remove(id)` is called while a `save(id)` is in flight. The concurrent-save × remove race is **completely untested**, and a leak would not be caught.
