# ERS Compliance Audit — nomad.md (Step 6 prep)

> Auditor: Compliance committee member
> Date: 2026-06-23
> Scope: Steps 1-5 + Step 6 readiness
> Spec: `docs/ers.md` v1.0.0
> Branch: `step-6-ui-audit`

---

## Verdict

The implementation is **largely on-spec** at the data + I/O + state layers, but **the UI layer has not been built** — Step 6 is the upcoming work and every visible behaviour in §4 (home screen, top bar, left rail, editor with Write/Preview tabs, Kanban, Gantt, theme toggle, filter bar) is currently a SvelteKit boilerplate placeholder (`src/routes/+page.svelte` is the default 2-line "Welcome to SvelteKit"). Step 6 is therefore the single biggest gap, but that is by design and acknowledged in `docs/current-project-status.md`. The non-UI halves of all 15 functional requirements (FR-1 … FR-15) are implemented and tested at the data layer; FR-2, FR-4, FR-5, FR-6, FR-7, FR-11, FR-13 (DOMPurify portion), and the entire UI surface of every FR are **DEFERRED**. NFR-13 (CSP / transport headers) is the only NFR that is structurally impossible without a build/deploy target and is **MISSING** by Step 6 design.

> **Spec discrepancy noted up-front:** the user's audit prompt requested scoring against "NFR-1 through NFR-15+". The authoritative spec (`docs/ers.md` v1.0.0) defines **only NFR-1 through NFR-7**. The scorecard below uses the spec as written.

> **Prior-audit note:** the prompt referenced `docs/requirements-audit.md`. That file does **not exist** in the repo. I have cross-checked against the available prior work — `docs/current-project-status.md`, `docs/security-audit-step-5.md`, `docs/senior-fullstack-review-step-5.md` — and call out the verification under §"What the prior audit got right / missed".

## Data model compliance

### §6.1 Issue file

- **Filename (§6.1.1)** — `buildIssueFilename(id, title)` produces `<padded id>-<slug>.md` (slugs.ts:35-37). `padIssueId` zero-pads to 4 digits (slugs.ts:27-29). `slugify` lowercases + collapses non-alphanumerics to `-` + trims (slugs.ts:16-24). Slugify has a defensive `'untitled'` fallback (slugs.ts:23) which the spec does not require — a small deviation that prevents filename collision on emoji-only titles.
- **Structure (§6.1.2)** — Parser splits `---` fence (frontmatter.ts:71-113) and walks body for `<!-- [SECTION_START: name] -->...<!-- [SECTION_END: name] -->` markers (parser.ts:24-46).
- **Frontmatter schema (§6.1.3)** — Every required key is parsed (parser.ts:147-165): `id`, `title`, `author`, `creation_date`, `updated_date`, `issue_type`, `status`, `assignee` (nullable), `labels` (array), `relations` (typed), `start_date`/`end_date` (nullable), `duration` (nullable number), `integrity_hash` (nullable), `customFields` (everything else). Field-to-YAML map is in `types/issue.ts:116-131`.
- **Sections (§6.1.4)** — Body MUST consist only of section blocks. The parser tolerates incidental outside-content by ignoring non-marker lines (parser.ts:42-43 comment); the spec says "MUST NOT contain any content outside of section blocks" but the parser is lenient. **Minor deviation:** the parser does not reject files that violate §6.1.4, but it does not surface the violation either. Acceptable as long as Step 6 displays a parse warning. Serializer emits sections in their stored order (no reorder — confirmed by `tests/services/integration.test.ts:374-419`).
- **Round-trip losslessness** — `tests/services/integration.test.ts:374-419` asserts every Issue field survives `serialize → write → read → parse`. MET.

### §6.2 Template file

- `assertTemplate` validates every required top-level key (template-loader.ts:13-77): `id`, `name`, `icon`, `color`, `default_status`, `fields` (array with typed entries), `sections` (array with typed entries).
- Field-type whitelist is `FIELD_TYPES` from `types/template.ts:14-23` — matches the spec exactly (`text`, `longtext`, `date`, `number`, `select`, `multi-select`, `user`, `relations`).
- `options` is validated to be an array when present (template-loader.ts:50-52). The spec says `options` is REQUIRED for `select`/`multi-select`; the loader does NOT enforce that — **a template with `type: 'select'` and no `options` will load without error**. Editor would then break. Step 6 + the wizard should ensure the loader (or wizard) rejects this.
- `options_source` is NOT validated by the loader — the loader merely accepts the field if present. Fine; the renderer is the consumer.
- Templates are sorted by `t.id.localeCompare` (template-loader.ts:111) — cosmetic, not spec-required.

### §6.3 Config file

- `assertConfig` validates every required field (config-loader.ts:10-57): `statuses`, `default_status`, `labels`, `users`, `kanban`, `gantt`, `remote`. Kanban must have `columns` array. Gantt must have `group_by` and `default_view` strings. Remote must have `cors_proxy` string.
- **Spec compliance:** all required keys per §6.3 are enforced. Optional fields (`kanban.columns` entries, `users` shape) are not deeply validated (e.g. an entry in `statuses` with a missing `id` would pass the loader and crash downstream). The validator (`services/validator.ts:139-141`) only checks `id` membership at issue-save time, not config-load time.
- **Example config (§6.3.1) would round-trip cleanly** — covered implicitly by `tests/services/config-loader.test.ts`.

### §6.4 Filename convention

- Already covered above. MET.

### §6.5 Trash convention

- Spec: `.nomad.md/.trash/<timestamp>-<id>-<slug>.md`.
- Implementation: `moveToTrash` (trash.ts:58-73) writes to `.nomad.md/.trash/<timestamp>-<uuid8>-<originalName>`. The original filename is preserved verbatim.
- **DEVIATION FROM SPEC:** the spec format uses the issue's own `id` and `slug` (`<timestamp>-<id>-<slug>.md`). The implementation uses a UUID-8 and the _original full filename_ (`<timestamp>-<uuid8>-<originalName>.md`). The implementation choice is _more_ collision-resistant (handles non-issue files, handles same-millisecond moves) but does not match the spec's literal string format. Documented as a deviation worth flagging in the audit; not a correctness defect (the spec format is sufficient for collision-avoidance in practice — UUID-8 is strictly stronger than the spec's reuse of `id`). Step 6's "Empty trash" UI listing will show the original filename at the tail, which is arguably nicer for the user.
- `emptyTrash` (trash.ts:87-99) clears only leaf files; subdirectories are preserved as a hardening choice. Covered by integration test `tests/services/integration.test.ts:281-300`.
